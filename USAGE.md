## 1. Local Development

### Prerequisites

- Node.js 20
- pnpm 9: `corepack enable && corepack prepare pnpm@9.0.0 --activate`
- Docker (for Postgres + Redis)
- `gcloud` CLI (for secret/GCS access during development)

### Start local services

```bash
# Postgres + Redis via Docker Compose (create this if you don't have it)
docker run -d --name postgres -p 5432:5432 \
  -e POSTGRES_USER=fatedworld_api \
  -e POSTGRES_PASSWORD=localdevpassword \
  -e POSTGRES_DB=fatedworld \
  postgres:15-alpine

docker run -d --name redis -p 6379:6379 redis:7-alpine
```

### Configure environment

```bash
cp apps/api/.env.example apps/api/.env
# Edit apps/api/.env:
#   DATABASE_URL="postgresql://fatedworld_api:localdevpassword@localhost:5432/fatedworld?schema=public"
#   REDIS_HOST=localhost
#   JWT_ACCESS_SECRET=any-random-string
#   JWT_REFRESH_SECRET=any-random-string
#   (leave GCS/Pub/Sub/Tasks blank — they log warnings but won't crash)
```

### Install and run

```bash
pnpm install

# Run database migrations
pnpm db:migrate          # runs prisma migrate deploy
pnpm db:generate         # generates Prisma client

# Start API (port 3000)
pnpm dev:api

# Start admin panel (port 3001) — in a second terminal
pnpm dev:admin
```

API is at `http://localhost:3000`
Swagger/OpenAPI docs: `http://localhost:3000/api/docs`
Admin panel: `http://localhost:3001`

### Create your first superadmin (local)

```bash
# Use Prisma Studio to edit the users table
pnpm db:studio
# Browse to users → set role = superadmin on your test user
# Or seed directly:
pnpm --filter @fated/api db:seed
```

## 2. Staging — First-Time Setup

This only needs to be done once. Most of it was already done by Terraform/Cursor, but the secrets need real values.

### 2a. Verify GCP staging is healthy

```bash
gcloud config set project fatedworld-staging
gcloud run services describe fated-api-staging --region=us-central1
# Should return the service URL
```

### 2b. Populate required Secret Manager secrets

These secrets exist in GCP but need **real values** inserted. Run each command:

**Anthropic API key** (used by distribution copy generation):
```bash
echo -n "sk-ant-..." | gcloud secrets versions add anthropic-api-key-staging \
  --data-file=- --project=fatedworld-staging
```

**FCM service account** (Firebase push notifications):
```bash
# Firebase Console → Project Settings → Service Accounts → Generate new private key
# Save the JSON file, then:
cat /path/to/firebase-sa.json | gcloud secrets versions add fcm-service-account-staging \
  --data-file=- --project=fatedworld-staging
```

**CDN signing key** (for signed video playback URLs):
```bash
# GCP Console → Network Services → Cloud CDN → fated-api-backend → URL signing keys
# Or create one:
gcloud compute backend-services create-signed-url-key fatedworld-key \
  --key-file=/tmp/cdn-key.bin --global 2>/dev/null || true
openssl rand -base64 32 | gcloud secrets versions add cdn-signing-key-staging \
  --data-file=- --project=fatedworld-staging
```

> The other secrets (`jwt-access-secret-staging`, `jwt-refresh-secret-staging`, `db-password-staging`, `internal-api-secret-staging`) were auto-generated by the bootstrap script and are already populated. You can verify with:
> ```bash
> gcloud secrets list --project=fatedworld-staging
> ```

### 2c. Run the initial migration

The staging CI runs migrations automatically on every push. To run manually:

```bash
gcloud run jobs execute fated-migrate-staging \
  --region=us-central1 --project=fatedworld-staging --wait
```

### 2c-fix. Grant CI service account IAM roles (required before first deploy)

The CI service account needs Secret Manager + Redis access. Run this once:

```bash
STAGING_PROJECT="fatedworld-staging"
STAGING_SA="fated-cloudbuild-sa@${STAGING_PROJECT}.iam.gserviceaccount.com"
for role in \
  roles/secretmanager.secretAccessor \
  roles/secretmanager.secretVersionManager \
  roles/redis.viewer \
  roles/iam.serviceAccountUser; do
  gcloud projects add-iam-policy-binding "${STAGING_PROJECT}" \
    --member="serviceAccount:${STAGING_SA}" \
    --role="${role}" --quiet
done
echo "Done. Re-trigger CI:"
gh workflow run staging.yml --repo thefatedworld/Fated
```

> These roles are now in Terraform too. Running `terraform apply` in `terraform/staging/` will make it permanent state.

### 2d. Verify staging is live

```bash
# Get the staging API URL
gcloud run services describe fated-api-staging \
  --region=us-central1 --project=fatedworld-staging \
  --format='value(status.url)'

# Health check (replace with your URL)
curl https://fated-api-staging-xxxx-uc.a.run.app/health
# Expected: {"status":"ok","db":"connected","redis":"connected"}
```

## 3. Staging — Day-to-Day Deploy

### Automatic deploy

Every push to `main` triggers the staging CI pipeline. It:
1. Runs tests and lint
2. Builds and pushes the API Docker image to Artifact Registry
3. Syncs the `database-url-staging` secret (reconstructs from `db-password-staging`)
4. Runs `prisma migrate deploy` via Cloud Run Job
5. Deploys the new API revision to Cloud Run (0% traffic initially)
6. Shifts 100% traffic to the new revision
7. Builds and deploys the admin panel

**Watch the deploy:**
```bash
# In GitHub: Actions tab → "Deploy to STAGING" workflow
# Or via CLI:
gh run list --repo thefatedworld/Fated
gh run watch <run-id>
```

### Manual deploy (skip CI)

```bash
gh workflow run staging.yml --repo thefatedworld/Fated
```

### Rollback staging

```bash
# List revisions
gcloud run revisions list --service=fated-api-staging \
  --region=us-central1 --project=fatedworld-staging

# Roll back to a specific revision
gcloud run services update-traffic fated-api-staging \
  --to-revisions=fated-api-staging-REVISION-ID=100 \
  --region=us-central1 --project=fatedworld-staging
```

## 4. PROD — First-Time Setup

### 4a. Create PROD GCP project

In GCP Console → New Project. Note the project ID (e.g. `fatedworld-prod`).

### 4b. Run the bootstrap script

```bash
export PROD_PROJECT_ID=fatedworld-prod
export GITHUB_ORG=thefatedworld
export GITHUB_REPO=Fated

bash scripts/setup-gcp-prod.sh
```

This script:
- Enables all required GCP APIs
- Creates Terraform state bucket
- Sets up Workload Identity Federation (keyless CI auth)
- Creates service accounts with correct IAM roles
- Auto-generates and stores all secrets
- Runs `terraform init` and `terraform plan` for PROD

At the end it prints the **GitHub secrets you must add**. Add them now:

```
GCP_PROD_PROJECT_ID           = fatedworld-prod
GCP_PROD_WIF_PROVIDER         = projects/123.../locations/global/workloadIdentityPools/...
GCP_PROD_SA_EMAIL             = fated-cloudbuild-sa@fatedworld-prod.iam.gserviceaccount.com
GCP_PROD_CLOUD_SQL_CONNECTION = fatedworld-prod:us-central1:fatedworld-prod
GCP_PROD_API_SA               = fated-api-sa@fatedworld-prod.iam.gserviceaccount.com
```

**Add them here:** `https://github.com/thefatedworld/Fated/settings/secrets/actions`

### 4c. Apply Terraform to PROD

```bash
cd terraform/prod
terraform apply \
  -var="project_id=fatedworld-prod" \
  -var="db_password=$(gcloud secrets versions access latest --secret=db-password-prod --project=fatedworld-prod)"
```

This creates: Cloud SQL (Postgres 15), Cloud Memorystore (Redis), VPC connector, Pub/Sub topics, Cloud Tasks queues, GCS buckets, Artifact Registry repo.

### 4d. Populate PROD secrets

Same as staging but with `-prod` suffix:

```bash
# Anthropic
echo -n "sk-ant-..." | gcloud secrets versions add anthropic-api-key-prod \
  --data-file=- --project=fatedworld-prod

# FCM
cat /path/to/firebase-sa.json | gcloud secrets versions add fcm-service-account-prod \
  --data-file=- --project=fatedworld-prod

# CDN signing key
openssl rand -base64 32 | gcloud secrets versions add cdn-signing-key-prod \
  --data-file=- --project=fatedworld-prod
```

### 4e. Add `production` environment to GitHub

`https://github.com/thefatedworld/Fated/settings/environments`

Create environment named `production`. Enable "Required reviewers" and add yourself. This gates the prod workflow on manual approval.

## 5. PROD — Deploying a Release

PROD deploys are **manual and gated**. You promote a staging image (identified by its git SHA) to prod.

### Step 1: Find the staging SHA to promote

```bash
# Get the SHA of the current staging image
gcloud run services describe fated-api-staging \
  --region=us-central1 --project=fatedworld-staging \
  --format='value(spec.template.metadata.annotations["run.googleapis.com/client-version"])'

# Or check GitHub Actions — note the SHA from the last successful staging deploy
gh run list --repo thefatedworld/Fated --workflow=staging.yml
```

Or just use the latest git SHA on main:
```bash
git rev-parse HEAD
```

### Step 2: Dispatch the prod workflow

```bash
gh workflow run prod.yml \
  --repo thefatedworld/Fated \
  -f image_tag=<FULL_40_CHAR_SHA> \
  -f confirm=deploy-to-prod
```

Or via GitHub UI: Actions → "Deploy to PRODUCTION" → Run workflow.

### Step 3: Approve the deployment

The workflow pauses at the `production` environment gate. Go to the GitHub Actions run and click **Approve**.

### What the prod workflow does

1. Pulls the API Docker image from **staging** Artifact Registry
2. Retags and pushes it to **prod** Artifact Registry
3. Syncs `database-url-prod` secret
4. Runs `prisma migrate deploy` against PROD database
5. Deploys the new revision to Cloud Run with **0% traffic** + a `canary-{sha}` tag
6. Shifts **10% traffic** to the canary revision, waits 5 minutes
7. Shifts **100% traffic** to latest
8. Deploys the admin panel to PROD

### Rollback PROD

```bash
gcloud run services update-traffic fated-api-prod \
  --to-revisions=fated-api-prod-PREVIOUS-REVISION=100 \
  --region=us-central1 --project=fatedworld-prod
```

## 6. Admin Panel

The admin panel is a Next.js app deployed to Cloud Run.

**Staging URL:** Get from:
```bash
gcloud run services describe fated-admin-staging \
  --region=us-central1 --project=fatedworld-staging \
  --format='value(status.url)'
```

**First login:** Register via the API, then use Prisma Studio or a direct DB query to set `role = 'superadmin'`:
```sql
UPDATE users SET role = 'superadmin' WHERE email = 'your@email.com';
```

### Admin panel capabilities

| Section | What you can do |
|---|---|
| **Content** | Create/edit series, seasons, episodes; schedule publishes; upload videos |
| **Video** | Upload episode assets (direct-to-GCS signed URL); manage asset versions |
| **Users** | Search users; view token history; change roles |
| **Moderation** | Review queue (wiki edits, abuse reports, refund escalations); ban/timeout users |
| **Analytics** | Platform KPI dashboard; episode performance; token economy |
| **Distribution** | Submit repackaging jobs; download outputs (video + AI copy) |
| **Wiki** | Approve/reject wiki revision submissions |

### Adding a series (example workflow)

1. Admin Panel → Content → Series → New
2. Fill title, slug, genre tags, cover image
3. Add Season → Add Episodes
4. For each episode: set `is_gated = true` and `token_cost` if paywalled
5. Upload video: Episode → Assets → Upload → use the signed URL to PUT the file
6. Set `status = scheduled` with a future `scheduled_at`, or publish immediately

### Local development

```bash
cd apps/mobile
pnpm install  # or: cd ../.. && pnpm install (from monorepo root)

# Set the API URL
# Edit apps/mobile/src/lib/api.ts (or wherever the base URL is configured)

pnpm ios      # iOS Simulator

```

### Building for TestFlight / internal testing

```bash
# Install EAS CLI
npm install -g eas-cli
eas login

# Configure (first time)
cd apps/mobile
eas build:configure

# Build for iOS (TestFlight)
eas build --platform ios --profile preview

# Build for Android (internal track)
eas build --platform android --profile preview
```

### Key app screens

| Screen | Description |
|---|---|
| Register/Login | Email + password; Apple/Google Sign-In (Phase 2) |
| Home | Recommendation feed + continue watching |
| Series detail | Cover, description, episodes list, community tab |
| Episode detail | Synopsis, token cost (if gated), unlock/play button |
| Video player | Signed URL playback; progress sync every 10s |
| Token wallet | Balance, purchase history, buy tokens (App Store/Play) |
| Community | Threads, episode comments, voting |
| Wiki | Lore pages, edit submissions |
| Profile | Settings, notification preferences, data export |

### IAP setup (required for token purchases)

**Apple:**
1. App Store Connect → In-App Purchases → Create consumables:
   - `tokens_100` ($0.99), `tokens_500` ($3.99), `tokens_1200` ($7.99), `tokens_3000` ($14.99)
2. App Store Connect → Server Notifications → set URL to:
   `https://your-api-url/v1/iap/apple/webhook`

**Google Play:**
1. Play Console → Monetization → In-app products → Create consumables (same IDs)
2. Play Console → Monetization → Real-time developer notifications → set Pub/Sub topic:
   `projects/fatedworld-staging/topics/iap-refund-staging`

## 8. Key API Endpoints

Base URL: `https://fated-api-staging-xxxx-uc.a.run.app` (staging)
Full OpenAPI spec: `{base_url}/api/docs`

### Auth

| Method | Path | Description |
|---|---|---|
| POST | `/v1/auth/register` | Register with email + password |
| POST | `/v1/auth/login` | Login → returns `{ accessToken, refreshToken }` |
| POST | `/v1/auth/refresh` | Rotate refresh token |
| POST | `/v1/auth/logout` | Revoke refresh token |

```bash
# Register
curl -X POST https://YOUR-API/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Password1!","username":"testuser"}'

# Login
curl -X POST https://YOUR-API/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"Password1!"}'
# → { "accessToken": "eyJ...", "refreshToken": "..." }
```

All protected endpoints require: `Authorization: Bearer {accessToken}`

### Content

| Method | Path | Description |
|---|---|---|
| GET | `/v1/series` | List series (paginated) |
| GET | `/v1/series/:slug` | Series detail |
| GET | `/v1/series/:seriesId/episodes` | Episodes list |
| GET | `/v1/episodes/:id` | Episode detail |
| GET | `/v1/episodes/:id/playback` | Get signed playback URL (requires entitlement) |

### Tokens & IAP

| Method | Path | Description |
|---|---|---|
| GET | `/v1/tokens/balance` | Current wallet balance |
| GET | `/v1/tokens/history` | Ledger history |
| POST | `/v1/iap/apple/validate` | Validate Apple StoreKit 2 JWS transaction |
| POST | `/v1/iap/google/validate` | Validate Google Play purchase token |
| POST | `/v1/iap/apple/webhook` | Apple refund webhooks (public, Apple-signed) |
| POST | `/v1/iap/google/webhook` | Google Play RTDN webhooks |

### Entitlements

| Method | Path | Description |
|---|---|---|
| POST | `/v1/episodes/:id/unlock` | Unlock an episode (debits tokens) |
| GET | `/v1/entitlements` | List user's unlocks |

### Community

| Method | Path | Description |
|---|---|---|
| GET | `/v1/community/threads` | List threads (filter by type/series/episode) |
| POST | `/v1/community/threads` | Create thread |
| POST | `/v1/community/threads/:id/replies` | Reply to thread |
| POST | `/v1/community/votes` | Upvote/downvote |

### Admin (superadmin/content_admin role required)

| Method | Path | Description |
|---|---|---|
| POST | `/v1/admin/series` | Create series |
| PATCH | `/v1/admin/series/:id` | Update series |
| POST | `/v1/admin/episodes` | Create episode |
| POST | `/v1/admin/episodes/:id/assets/upload-url` | Get GCS signed upload URL |
| GET | `/v1/admin/moderation/queue` | Review queue |
| POST | `/v1/admin/moderation/actions` | Ban, timeout, delete content |

### Analytics

| Method | Path | Description |
|---|---|---|
| POST | `/v1/analytics/events` | Ingest playback/engagement events |
| GET | `/v1/author/series/:id/analytics` | Author's series analytics |
| GET | `/v1/admin/analytics/dashboard` | Platform KPI snapshot |

## 9. Operations Reference

### View logs

```bash
# API logs (live tail)
gcloud logging tail "resource.type=cloud_run_revision AND resource.labels.service_name=fated-api-staging" \
  --project=fatedworld-staging

# Errors only
gcloud logging read "severity>=ERROR AND resource.labels.service_name=fated-api-staging" \
  --project=fatedworld-staging --limit=50 --format=json | jq '.[].jsonPayload'
```

### DB access (staging)

```bash
# Install Cloud SQL Auth Proxy
curl -o cloud-sql-proxy https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.14.1/cloud-sql-proxy.darwin.amd64
chmod +x cloud-sql-proxy

# Connect
./cloud-sql-proxy --port=5433 fatedworld-staging:us-central1:fatedworld-staging &

# Psql
DB_PASS=$(gcloud secrets versions access latest --secret=db-password-staging --project=fatedworld-staging)
psql "postgresql://fatedworld_api:${DB_PASS}@localhost:5433/fatedworld"

# Or use Prisma Studio (reads from apps/api/.env)
pnpm db:studio
```

### Run a migration manually

```bash
# Staging
gcloud run jobs execute fated-migrate-staging \
  --region=us-central1 --project=fatedworld-staging --wait

# PROD
gcloud run jobs execute fated-migrate-prod \
  --region=us-central1 --project=fatedworld-prod --wait
```

### Emergency: disable IAP validation

If there's a suspected double-credit bug, temporarily disable IAP:
```bash
gcloud run services update fated-api-staging \
  --update-env-vars=IAP_VALIDATION_ENABLED=false \
  --region=us-central1 --project=fatedworld-staging
# Fix the bug, then re-enable:
gcloud run services update fated-api-staging \
  --remove-env-vars=IAP_VALIDATION_ENABLED \
  --region=us-central1 --project=fatedworld-staging
```

### Redis inspection

```bash
# Connect to Redis (requires VPC access or Cloud Shell)
gcloud redis instances describe fated-redis-staging \
  --region=us-central1 --project=fatedworld-staging \
  --format='value(host)'
# Use the IP from Cloud Shell within the same project
```

### BigQuery — run analytics manually

```bash
# Open Looker Studio with the BigQuery dataset:
# https://lookerstudio.google.com → Connect → BigQuery → fatedworld-staging → fatedworld_analytics

# Or query directly:
bq query --project_id=fatedworld-staging \
  'SELECT event_type, COUNT(*) as cnt FROM fatedworld_analytics.events_raw
   WHERE DATE(_PARTITIONTIME) = CURRENT_DATE()
   GROUP BY 1 ORDER BY 2 DESC LIMIT 20'
```

### Monitoring alerts

Cloud Monitoring is configured to alert on:
- Error rate > 1% (P1 alert)
- p99 latency > 2s
- Cloud SQL connections > 80%
- Uptime check failure on `/health`

**View alerts:** GCP Console → Monitoring → Alerting → `fatedworld-staging`

### Update a secret value

```bash
echo -n "new-value" | gcloud secrets versions add SECRET_NAME \
  --data-file=- --project=fatedworld-staging
# Cloud Run picks up the new version on next request (no restart needed for :latest)
```

### Check GitHub Actions secrets are set

```bash
gh secret list --repo thefatedworld/Fated
# Should show: GCP_STAGING_PROJECT_ID, GCP_STAGING_WIF_PROVIDER,
#              GCP_STAGING_SA_EMAIL, GCP_STAGING_CLOUD_SQL_CONNECTION,
#              GCP_STAGING_API_SA
# (Plus GCP_PROD_* after prod setup)
```
